### Struttura del file system
il file system risiede in una partizione di memoria, abbiamo due approcci per gestire le partizioni del disco:
- Master Boot Record (MBR)
- GUID Partion Table

Perché usare le partizioni è importante? perché riusciamo ad avere una divisione logica del disco che ci permette anche la creazione del dual boot

###### MBR
All'interno del disco viene riservata una piccola locazione di memoria contiene 4 voci una per per partizione dove vengono specificate delle informazioni importante ai file system che girano all'interno delle partizioni

Questo metodo è un metodo storico che supporta il partizionamento di dischi che hanno una dimensione inferiore a 2TB

### Implementazione dei file
Ci sono diversi modi per farlo:
- **Allocazione contigua**: ha tanti difetti, gli stessi legati all'allocazione contigua di memoria centrale (Cosa succede se uno specifico blocco si espande e tocca il successivo? Se non c'è una locazione di memoria abbastanza grande? frammentazione esterna, frammentazione interna, ecc...)
- **Allocazione con liste collegate**: approccio banale, ogni blocco è collegato al blocco successivo/precedente (La ricerca di un blocco dentro $n$ blocchi dedicati ad un singolo file memoria è $O(n)$ troppo lento per un file system)
- **FAT**
- **I-Node**: una struttura dati associata al file, al suo interno vengono rappresentate delle informazioni (metadati). Questi I-node sono memorizzati in una struttura dati (una tabella in pratica). Mi permettono di allocare file in maniera non contigua.
